## F1 ŌĆö User Management

| ID     | Feature                              | Priority | Status | Description |
|--------|--------------------------------------|----------|--------|-------------|
| F1.1   | User registration & authentication   | High     | Done   | Players register with email, first name, last name, screen name, and player ID (Pokemon TCG player ID). Email must be verified via a token-based activation link before the account becomes active. Global roles: player, organizer, admin. Staff is a per-event assignment (see F3.5). First name and last name are used for the public "FirstName L." display in tournament results (F3.17); the screen name is the primary display name for authenticated views. **Target path redirect:** login and register links across the app pass the current page as `_target_path` so users return to their original page after authentication instead of always landing on the dashboard. | F1.3   | User profile                         | Medium   | Done   | Display screen name, player ID, owned decks, borrow history, and upcoming event participation. |
| F1.4   | Role-based access control            | High     | Done   | Global roles: admin, organizer, cms_editor, archetype_editor, player. Staff is a **per-event assignment**, not a global role (see F3.5). Admin can manage users and has all permissions. | F1.7   | Password reset                       | High     | Done   | Standard forgot-password flow: user requests a reset link by email, receives a tokenized URL, and sets a new password. | F1.8   | Account deletion & data export (GDPR) | Medium  | Done   | User can request account deletion from their profile. | F1.9   | Login with screen name or email | Low |        | The login form accepts either the user's email or screen name as identifier. `UserRepository::loadUserByIdentifier()` queries by email OR screenName. The form field label updates to "Email or screen name". Depends on F1.1. |
| F1.10  | Password strength scoring (zxcvbn) | Low |        | Passwords are scored using the zxcvbn algorithm (0ŌĆō4) on both frontend and backend. **Backend (PHP):** a `PasswordStrengthScorer` service using `bjeavons/zxcvbn-php` scores the password on registration, password reset, and profile update; the score is stored on `User.passwordScore` (nullable integer 0ŌĆō4). User-specific dictionary (email, screenName, firstName, lastName) is injected to penalize passwords derived from personal data. **Frontend (JS):** `zxcvbn` (npm) powers a real-time strength gauge (color-coded meter + label) on registration, password reset, and profile update forms with the same user dictionary. The gauge is advisory ŌĆö the backend enforces minimum length (8 chars) but does not reject by score. Depends on F1.1. |
| F1.11  | Gravatar avatar & navbar user menu | Medium | Done   | User avatar powered by **Gravatar**: the avatar URL is built from the MD5 hash of the user's lowercase trimmed email (`https://www.gravatar.com/avatar/{hash}?d=mp&s=80`). The `mp` (mystery-person) default serves as fallback when no Gravatar is configured. A **round avatar thumbnail** (32 px) is displayed in the top navbar alongside the user's screen name. Clicking the avatar opens a **dropdown menu** with links to: User profile (F1.3), Gravatar settings (external link to gravatar.com), and Logout. When the user is not authenticated, a generic person icon is shown instead, linking to the login page. A Twig extension (`GravatarExtension`) exposes a `gravatar(email, size)` function for reuse across templates. Depends on F1.1. |
| F1.12  | Play Pokemon QR scan for player identification | Medium |        | Scan a player's **Play! Pokemon access QR code** (displayed in the Pokemon TCG Live app or on the Pokemon website) using the device camera (F5.6) or a USB HID scanner (F5.3). The QR code contains a **JWT token** with player identity data (name, player ID, etc.). The system decodes and validates the JWT, then uses the extracted data in three contexts: **(1) Quick account creation** ŌĆö an admin scans a player's QR to pre-fill a new account (first name, last name, Pokemon ID) and create it without email verification, with a temporary password the player must change on first login. **(2) Staff assignment** ŌĆö an organizer scans the QR on the event page to instantly identify and assign a player as event staff (F3.5), replacing manual search. **(3) Player identification** ŌĆö at an event, staff scans a player's QR to quickly look up their profile, active borrows, and deck assignments. The JWT parsing is handled by a dedicated `PlayPokemonQrDecoder` service. Depends on F1.1, F5.3 or F5.6. |

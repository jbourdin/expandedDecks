# Feature List

The frontend is built with **React.js** (via Symfony UX / Webpack Encore) for all interactive UI components.

## F1 — User Management

| ID     | Feature                              | Priority | Description |
|--------|--------------------------------------|----------|-------------|
| F1.1   | User registration & authentication   | High     | Players register with email, first name, last name, screen name, and player ID (Pokemon TCG player ID). Email must be verified via a token-based activation link before the account becomes active. Global roles: player, organizer, admin. Staff is a per-event assignment (see F3.5). First name and last name are used for the public "FirstName L." display in tournament results (F3.17); the screen name is the primary display name for authenticated views. **Target path redirect:** login and register links across the app pass the current page as `_target_path` so users return to their original page after authentication instead of always landing on the dashboard. The target path is saved in the session (via `TargetPathTrait`) and validated by `SafeAuthenticationSuccessHandler` to prevent open-redirect attacks (only relative paths without protocol are accepted). |
| F1.2   | Email verification                   | High     | On registration, a verification token is sent by email. The account remains inactive until the user clicks the activation link. Token expires after a configurable delay. |
| F1.3   | User profile                         | Medium   | Display screen name, player ID, owned decks, borrow history, and upcoming event participation. |
| F1.4   | Role-based access control            | High     | Global roles: admin, organizer, cms_editor, archetype_editor, player. Staff is a **per-event assignment**, not a global role (see F3.5). Admin can manage users and has all permissions. Organizers can create events and assign staff. **CMS editors** can create, edit, and publish content pages and menu categories (F11.1, F11.2). **Archetype editors** can create, edit, and publish archetype descriptions (F2.6). Players can register decks and request borrows. See [User model — Roles](models/user.md). |
| F1.5   | MFA with TOTP (planned)              | Low      | Multi-factor authentication using TOTP (Google Authenticator, Authy, etc.). Not in initial release — planned for a future iteration. |
| F1.6   | Pokemon SSO (to investigate)         | Low      | Investigate feasibility of integrating Pokemon Company SSO for seamless player identification. Requires outreach to The Pokemon Company to assess API availability and authorization. |
| F1.7   | Password reset                       | High     | Standard forgot-password flow: user requests a reset link by email, receives a tokenized URL, and sets a new password. Mirrors the existing verification token pattern (`verificationToken` / `tokenExpiresAt`). Only one active reset token per user at a time. |
| F1.8   | Account deletion & data export (GDPR) | Medium  | User can request account deletion from their profile. Soft-delete with **anonymization**: personal data (email, screenName, playerId) replaced with anonymous placeholders, borrow/event history preserved for data integrity. Confirmation via **email link** (expires 24 h). Users can also **export all personal data as JSON** (profile, borrows, events, decks). See [User model — GDPR](models/user.md). |
| F1.9   | Login with screen name or email | Low | The login form accepts either the user's email or screen name as identifier. `UserRepository::loadUserByIdentifier()` queries by email OR screenName. The form field label updates to "Email or screen name". Depends on F1.1. |
| F1.10  | Password strength scoring (zxcvbn) | Low | Passwords are scored using the zxcvbn algorithm (0–4) on both frontend and backend. **Backend (PHP):** a `PasswordStrengthScorer` service using `bjeavons/zxcvbn-php` scores the password on registration, password reset, and profile update; the score is stored on `User.passwordScore` (nullable integer 0–4). User-specific dictionary (email, screenName, firstName, lastName) is injected to penalize passwords derived from personal data. **Frontend (JS):** `zxcvbn` (npm) powers a real-time strength gauge (color-coded meter + label) on registration, password reset, and profile update forms with the same user dictionary. The gauge is advisory — the backend enforces minimum length (8 chars) but does not reject by score. Depends on F1.1. |
| F1.11  | Gravatar avatar & navbar user menu | Medium | User avatar powered by **Gravatar**: the avatar URL is built from the MD5 hash of the user's lowercase trimmed email (`https://www.gravatar.com/avatar/{hash}?d=mp&s=80`). The `mp` (mystery-person) default serves as fallback when no Gravatar is configured. A **round avatar thumbnail** (32 px) is displayed in the top navbar alongside the user's screen name. Clicking the avatar opens a **dropdown menu** with links to: User profile (F1.3), Gravatar settings (external link to gravatar.com), and Logout. When the user is not authenticated, a generic person icon is shown instead, linking to the login page. A Twig extension (`GravatarExtension`) exposes a `gravatar(email, size)` function for reuse across templates. Depends on F1.1. |
| F1.12  | Play Pokemon QR scan for player identification | Medium | Scan a player's **Play! Pokemon access QR code** (displayed in the Pokemon TCG Live app or on the Pokemon website) using the device camera (F5.6) or a USB HID scanner (F5.3). The QR code contains a **JWT token** with player identity data (name, player ID, etc.). The system decodes and validates the JWT, then uses the extracted data in three contexts: **(1) Quick account creation** — an admin scans a player's QR to pre-fill a new account (first name, last name, Pokemon ID) and create it without email verification, with a temporary password the player must change on first login. **(2) Staff assignment** — an organizer scans the QR on the event page to instantly identify and assign a player as event staff (F3.5), replacing manual search. **(3) Player identification** — at an event, staff scans a player's QR to quickly look up their profile, active borrows, and deck assignments. The JWT parsing is handled by a dedicated `PlayPokemonQrDecoder` service. Depends on F1.1, F5.3 or F5.6. |

## F2 — Deck Library

| ID     | Feature                              | Priority | Description |
|--------|--------------------------------------|----------|-------------|
| F2.1   | Register a deck                      | High     | A user registers a physical deck they own, assigning it a name, archetype, and format (Expanded). On creation, a **short tag** is auto-generated: a 6-character unique code (uppercase letters + digits, excluding `I`/`O` to prevent confusion with `1`/`0` — allowed charset: `A-H, J-N, P-Z, 0-9`, 34 chars, 34^6 ≈ 1.5 B possible codes). The short tag serves as a **verbal identifier** at events ("Can I borrow deck AB3K7N?"), a **searchable field** in the app (catalog, borrow flows), and is **printed on the deck box label** (F5.x). See [Deck model — shortTag](models/deck.md). |
| F2.2   | Import deck list (copy-paste)        | High     | User pastes a deck list in standard PTCG text format. The system parses it (`ptcgo-parser`), validates each card against TCGdex, checks Expanded legality (Black & White onward + banned list), and creates a new `DeckVersion` with the parsed cards. The raw text is preserved on the version for reference. `Deck.currentVersion` is updated to the newly created version. |
| F2.3   | Deck detail view                     | Medium   | Display deck info: owner, deck's archetype and card list (categorized: Pokemon / Trainer by subtype / Energy, sorted by quantity then name), availability status, languages, borrow history, and version history with the ability to view past versions. Mouse over a card name shows the card image (from TCGdex). **Public decks** (see `Deck.public`) are accessible to anonymous visitors via a shortTag URL (`/deck/{shortTag}`); non-public decks require authentication and ownership, admin role, or event-staff relationship. Anonymous visitors see login/register CTAs in the borrow section and are redirected back to the deck page after authentication (via `_target_path`). |
| F2.4   | Deck catalog (browse & search)       | Medium   | List all registered **public** decks (see `Deck.public`). Filters: free-text **search** (matches deck name and shortTag), **archetype** (Mantine autocomplete via `/api/archetype/search`), **event** (Mantine autocomplete via `/api/event/search` — show only decks that were played at a specific event via F3.7), **owner** (Mantine autocomplete via `/api/deck-owner/search`). All filter endpoints are publicly accessible (no authentication). Default sort: **best tournament result** (lowest placement ascending — a 1st-place finish ranks above 2nd), then by **`Deck.updatedAt` descending** (most recently updated first). Decks without any tournament result appear after those with results, sorted by `updatedAt` only. The catalog is publicly accessible at `/deck` (anonymous visitors can browse) and serves as the full **deck library**. Paginated (12 per page). Authenticated users see a "New Deck" CTA button. |
| F2.5   | Deck availability status             | High     | Each deck has a real-time status: available, lent, reserved, retired. |
| F2.6   | Deck archetype management            | Low      | A managed catalogue of archetypes (e.g. "Lugia VSTAR", "Mew VMAX"). Initial implementation: each `Archetype` has a **name** (display name, unique) and a **slug** (URL-friendly, auto-generated, unique). Deck owners select an archetype from the managed list when registering or editing a deck; free-text input with autocomplete suggests existing archetypes, and users can dynamically create new archetypes. Archetype lives on the **Deck** entity (not DeckVersion). Future enhancements: `pokemonSlugs` for sprite pictograms (F2.12), translatable fields (name, description, metaDescription), `isPublished` flag, `ROLE_ARCHETYPE_EDITOR` management. See [Archetype entity in Deck model](models/deck.md). |
| F2.7   | Retire / reactivate a deck           | Low      | Owner can mark a deck as retired (no longer available) or reactivate it. |
| F2.8   | Update deck list (new version)       | High     | Owner pastes an updated deck list → creates a new `DeckVersion`. `Deck.currentVersion` moves to the new version. Previous versions are preserved for history. Estimated value can be updated per version. Languages and archetype are deck-level properties (edited via the deck edit form). |
| F2.9   | Deck version history                 | Medium   | View all past versions of a deck: version number, archetype, creation date, and card list. Compare versions to see what changed (cards added/removed/quantity changed). |
| F2.10  | Archetype detail page                | Low      | Public page at `/archetypes/{slug}` displaying the archetype **name**, **sprites** (F2.12), and **description** rendered from Markdown with custom tag expansion. Three custom tags are supported: `[[archetype:slug]]` renders as a linked archetype name with sprites, `[[deck:SHORTTAG]]` renders as a badge link to the deck page, and `[[card:SET-NUMBER]]` renders as the card name with a hover/tap image preview (resolved from local DB or TCGdex API, cached 24h). The full rendered output is cached for 1 hour keyed by content hash. Includes a **"Browse decks"** CTA linking to the deck catalog (F2.4) pre-filtered by this archetype with available deck count. Unpublished archetypes return 404 (visible to admins only with draft notice). Card hover reuses the shared `card-hover.ts` module (extracted from deck show). SEO meta tags populated from `metaDescription`. |
| F2.11  | Archetype backlinking                | Low      | Wherever an archetype name appears in the application — deck detail (F2.3), deck catalog (F2.4), tournament results (F3.17), event available decks, and admin archetype list (F2.6) — it is rendered as a **hyperlink** to the archetype detail page (F2.10). Unpublished archetypes are displayed as plain text instead (except in the admin list which always links). |
| F2.15  | Archetype playstyle tags             | Low      | Archetypes can be tagged with one or more **free-text playstyle tags** (e.g. Aggressive, Control, Combo, Lock, Spread, Toolbox). Stored as a JSON array of strings on the `Archetype` entity (`playstyleTags` column). Tags are **normalized** on save: title case, alphanumeric and spaces only, deduplicated. The admin form uses a Mantine **TagsInput** component that suggests existing tags (collected from all archetypes) and allows creating new ones by typing. Displayed as `bg-secondary` badges in the archetype detail page header (F2.10) and admin archetype list (F2.6). Managed via the admin archetype form (F2.18). Enables future filtering in the deck catalog (F2.4) and archetype catalog (F2.16). |
| F2.17  | Deck catalog archetype filter UX     | Medium   | Replace the current empty archetype text input in the deck catalog (F2.4) with a **searchable dropdown** populated from published archetypes. The dropdown should display archetype names with sprites, support type-ahead filtering, and be implemented using a Mantine `Select` or `Combobox` component (or equivalent). Only published archetypes with at least one public deck should appear. When no archetype is selected the filter is hidden or shows a placeholder — never an empty input field. |
| F2.16  | Archetype catalog                    | Medium   | Public browsable page listing all published archetypes with sprites, names, and playstyle tags (F2.15). **Filterable** by playstyle tags. **Sortable** by: number of available decks, popularity (bookmark count, see F13.3), and recent tournament performance (results from the last 6 months). Each entry links to the archetype detail page (F2.10). Serves as the main entry point for players exploring which archetypes are available to borrow. |
| F13.3  | Bookmark an archetype                | Low      | Users can bookmark archetypes for quick access. Bookmark count is used as a popularity signal for sorting in the archetype catalog (F2.16). |
| F2.18  | Admin archetype create/edit form     | Medium   | **Create** button on the admin archetype list (`/admin/archetypes`) links to `/admin/archetypes/new`. The form (`ArchetypeFormType`) includes: name, description (Markdown textarea), metaDescription, pokemonSlugs (comma-separated text input synced to hidden JSON field via inline JS), playstyle tags (Mantine `TagsInput` with existing tag suggestions and free-text creation, F2.15), and isPublished toggle. Slug is auto-generated from name via `AsciiSlugger` on persist/update. The same form type is used for both create and edit. Create persists a new `Archetype` entity; edit flushes changes. Flash messages confirm success. Accessible to `ROLE_ADMIN` users. |
| F2.13  | Inline deck list import on creation  | Medium   | On the deck creation form, add an optional **pastable deck list** textarea. When filled, the deck is created and the list is immediately imported in one step — combining F2.1 and F2.2 into a single action. If left empty, only the deck metadata is saved (existing behavior). |
| F2.14  | Deck event status overview           | Medium   | On the deck detail page (F2.3), display a section visible to the **deck owner** listing all future events where the owner has at least an `interested` engagement state (F3.13). For each event, show the deck's status: registered (via `EventDeckRegistration`), delegated to staff, played (via `EventDeckEntry`), actively borrowed, or not registered. Provides a quick overview of the deck's commitments across upcoming events. |
| F2.12  | Archetype sprite pictograms          | Low      | Each archetype displays one or more **Pokémon box sprites** as a visual pictogram, rendered from the `Archetype.pokemonSlugs` array via the `archetype_sprites()` Twig function. Sprites are sourced from a **[PokéSprite fork with Gen 9 support](https://github.com/martimlobao/pokesprite)** (1478 box sprite PNGs, native sizes 23×22 to 73×62 px, covers Gen 1–9 including Scarlet/Violet and DLC Pokémon). Sprites are **downloaded at build time** via `make sprites` (tarball cached in gitignored `assets/vendor/sprites/pokemon/`) and copied to `public/build/sprites/pokemon/` by `copy-webpack-plugin` during `make assets` — no runtime CDN dependency, no sprites committed to git. Sprites are displayed **between the short tag badge and the deck name** wherever a deck appears: catalog (F2.4), deck detail (F2.3), dashboard (F7.1), event pages (F3.3), borrow views (F4.x), tournament results (F3.17), and the admin archetype list (F2.6). Rendered at a fixed **40px CSS height** with `image-rendering: pixelated` for crisp pixel art scaling; width scales proportionally. `alt` and `title` attributes use a human-readable name derived from the slug (e.g. `iron-thorns` → `Iron Thorns`). In table views, a `.archetype-sprites` wrapper provides `min-width` for consistent name alignment across 1–3 sprites. When `pokemonSlugs` contains multiple entries (e.g. `["roaring-moon", "flutter-mane"]`), sprites are rendered side by side. PokéSprite is credited in the documentation ([Credits](credits.md)) and in the `/legal` CMS page (F11.1). |

## F3 — Event Management

| ID     | Feature                              | Priority | Description |
|--------|--------------------------------------|----------|-------------|
| F3.1   | Create an event                      | High     | An organizer (or admin) declares an upcoming event with name, event ID, date, location, format, registration link, tournament structure (`TournamentStructure` enum), entry fee (cents + ISO 4217 currency), min/max attendees, round durations, and decklist requirement. The tournament ID field has a **"Sync" CTA** to prefill the form from the official Pokemon event page (F3.18). The form displays a **localized philosophy notice** (`app.event.philosophy_notice`) reminding the organizer that this tool is an independent community project, does not replace official services, and that official tournament rules must be followed (see [Philosophy](philosophy.md)). |
| F3.2   | Event listing                        | Medium   | Browse upcoming and past events with date, location, and participant count. **Publicly accessible** (no authentication required) — the anonymous homepage (F10.2) links here as the primary entry point for browsing events. For authenticated users, the **homepage event list** defaults to showing only events where the current user has an engagement state of at least `interested` (see F3.13). A **"Find Tournament"** call-to-action is displayed next to the list, linking to the event discovery page (F3.15). |
| F3.3   | Event detail view                    | Medium   | Show event info (tournament structure, entry fee, registration link, round durations, attendee limits, decklist requirement), list of borrow requests, and deck assignments for that event. |
| F3.4   | Register participation to an event   | Medium   | A player declares they intend to attend an event (prerequisite to requesting a borrow). Participation has two modes: **playing** (the player intends to compete and may borrow a deck) or **spectating** (the player attends without playing — they can still lend their own decks to others but cannot borrow). Spectating is useful for mixed-format events where the player may not qualify for a later stage (e.g. Standard day two or top cut) or simply wants to support the community by making decks available. Changing mode (spectating ↔ playing) is allowed as long as the event hasn't ended. |
| F3.5   | Assign event staff team              | High     | An organizer assigns staff members to an event. Staff role is **per event** (not a global role). Staff can then act as intermediaries for deck lending at that event only. The staff assignment input supports **multi-field search** (screen name, email, or Pokemon ID) with **autocomplete** suggestions via a JSON API endpoint (`GET /api/user/search?q=`). |
| F3.6   | Tournament ID verification (to investigate) | Low | Investigate whether the Pokemon tournament system exposes an API to verify that the organizer is the actual TO of the referenced tournament ID. Related: F3.18 already fetches the event page and can extract the organizer name — this could serve as a lightweight verification hint. |
| F3.7   | Register played deck for event       | Medium   | A player (deck owner or borrower) records which deck version they played at an event, creating an `EventDeckEntry`. This is separate from borrowing — it tracks tournament deck registration for history and traceability. After the event, the organizer or staff can record the player's **final placement** (integer ranking, e.g. 1st, 2nd, top 4, top 8) and **match record** (wins-losses-ties). These results feed into the tournament results page (F3.17) and the deck library sort order (F2.4). |
| F3.9   | Edit an event                        | High     | An organizer can update an event's details (name, date, location, tournament structure, entry fee, attendee limits, round durations, registration link, decklist requirement) as long as the event has not ended. The **"Sync" CTA** (F3.18) is also available on the edit form to re-fetch data from the Pokemon event page. The same **philosophy notice** as F3.1 is displayed. Participants are notified of material changes (date, location, cancellation — see F8.2). |
| F3.10  | Cancel an event                      | Medium   | An organizer can cancel an event. Cancellation cascades to **pre-handoff borrows only**: all `pending` and `approved` borrows for this event are automatically cancelled (deck status reverts to `available`), and affected borrowers/owners are notified (F8.2). **Already-lent decks are not affected** — borrows in `lent` (or `overdue`) status remain active because the physical deck is out and must still be returned by the borrower; deck owners are notified that the event was cancelled but their deck is still out. Cancelled events remain visible (for history) but are clearly marked. A cancelled event cannot be un-cancelled or edited further. |
| F3.11  | Event visibility                     | Medium   | Each event has a visibility setting controlling discoverability. Two modes: **public** — visible to all visitors (including unauthenticated users), listed in the event finder (F3.15) and agendas; **invitation-only** — visible only to users who have been explicitly invited by the event organizer or staff (the `invited` engagement state from F3.13 doubles as access grant). Default: public. The event listing (F3.2), event finder (F3.15), and detail view (F3.3) all respect this setting. Organizers, event staff, and admins always see all events regardless of visibility. |
| F3.12  | Event tags                           | Low      | Tag events with freeform labels (e.g. "June Expanded Cup", "Île-de-France League", "Weekly Thursday"). Each event can have **multiple tags**. Tags are **searchable** from the event listing (F3.2) and event finder (F3.15) — selecting a tag filters the list to matching events. Each tag has a **shareable URL** (e.g. `/events?tag=june-expanded-cup`) that can be bookmarked or shared. The tag input field on event create/edit (F3.1, F3.9) provides **autocomplete suggestions** from existing tags and allows creating new tags inline. Tags are stored as a dedicated `EventTag` entity with a unique slug derived from the label. Tag management (rename, merge, delete) is available to admins. |
| F3.13  | Player engagement states             | Medium   | A player-event relationship goes through engagement states that determine agenda visibility and access. States: **interested** — the player self-declares interest; the event appears in their personal agenda and iCal feed (F3.14), but they are not yet committed to attending. **invited** — set by event organizer or staff only; grants visibility for invitation-only events (F3.11) and adds the event to the player's agenda. **registered (playing)** — the player has confirmed participation and intends to compete (existing F3.4 behavior). **registered (spectating)** — the player has confirmed attendance as a spectator (see F3.4). Progression: a player can move from `interested`/`invited` → `registered (playing/spectating)` and back to `interested` if they un-register. The `invited` state can only be added by event staff or the organizer but can be combined with any registration state. The homepage event list (F3.2) defaults to showing events where the player has any engagement state. |
| F3.14  | iCal agenda feed                     | Low      | Each user has a personal, token-authenticated iCal/ICS feed URL (e.g. `/ical/{userToken}.ics`). The feed contains all events where the user has any engagement state (`interested`, `invited`, `registered`). Each event is rendered as a `VEVENT` with name, date/time, location, and a link to the event detail page. The feed updates dynamically. The token is regenerable from the user profile (invalidates the previous URL). Useful for syncing with Google Calendar, Apple Calendar, Outlook, etc. |
| F3.15  | Event discovery                      | Medium   | A dedicated **"Find Tournament"** page for browsing and searching public events (visibility = `public`). Filters: date range, format, location/region, tournament structure, tag (F3.12). Results show event name, date, location, participant count, and a quick "I'm interested" action (sets the `interested` state from F3.13). Accessible from the homepage CTA next to the user's event list (F3.2). Also serves as the landing page for unauthenticated visitors exploring available events. |
| F3.16  | Public iCal feed                     | Low      | An anonymous, unauthenticated iCal/ICS feed at a fixed URL (e.g. `/ical/public.ics`). Contains all events with `public` visibility (F3.11). Past events are **truncated to 1 month** (events whose `endDate` — or `date` for single-day events — is older than 1 month are excluded). Future events have no cutoff. The response sets `Cache-Control: public, max-age=3600` (1-hour HTTP cache). Each event is rendered as a `VEVENT` with name, date/time, location, tournament structure, and a link to the event detail page. Cancelled events are excluded. No authentication or token required — suitable for embedding in community calendars and public websites. |
| F3.17  | Tournament results                   | Medium   | Public page showing the results of a completed event. Displays the final standings: placement, player name, match record (W-L-T), deck archetype, and a link to the played deck version (F3.7). **Privacy:** in public/anonymous view, player names are displayed as **"FirstName L."** (first name + first letter of last name) using the `firstName`/`lastName` fields from the User model. Authenticated users see the player's full screen name instead. Results are ordered by placement ascending (1st, 2nd, …). Each event detail page (F3.3) links to its results once results have been entered. Only visible for events with `public` visibility. |
| F3.18  | Sync from Pokemon event page         | Medium   | A **"Sync"** button next to the Tournament ID field on the event create (F3.1) and edit (F3.9) forms. When the user enters a tournament ID (e.g. `26-03-000114`) and clicks Sync, the system fetches the official Pokemon event page (`https://www.pokemon.com/{locale}/play-pokemon/pokemon-events/{tournamentId}/`) via a lightweight **backend proxy endpoint** (needed to bypass CORS). The proxy parses the page's **JSON-LD** `schema.org/Event` markup and key HTML elements, then returns structured JSON. Available fields: event name, start date, venue name, full address (street, city, region, postal code, country), format (e.g. "TCG: Standard" → mapped to app format values), event type (e.g. "League Cup", "Regional", "Special Event" → mapped to `TournamentStructure` enum), and organizer name. The **frontend receives the JSON and prefills the form** — this is purely a helper; the user reviews, can modify any field, and explicitly saves. Fields already filled by the user are **not overwritten** unless the user confirms. Visual feedback: loading spinner on the Sync button, success toast listing prefilled fields, or error message if the page is unreachable or the tournament ID is invalid. |
| F3.19  | League deduction from Pokemon ID (to investigate) | Low | Investigate whether a user's league affiliation can be deduced from their Pokemon ID via official Play! Pokemon APIs. League features should only be added if data can be sourced from official systems rather than managed locally. Related: F3.18 already extracts venue/organizer data from the Pokemon event page. |
| F3.20  | Mark event as finished                   | Medium   | The organizer marks the event as **finished** at any time (no end-date restriction). Sets `Event.finishedAt`. Triggers: **(1)** overdue reminders to borrowers of unreturned decks (email + in-app per F8.3 preferences), **(2)** closes event — no new borrows, registrations, or edits, **(3)** unlocks tournament result entry (F3.17). A finished event cannot be un-finished. Distinct from F3.10 (cancellation): cancelled = event didn't happen; finished = event completed normally. See [Event model — Finishment Behavior](models/event.md). Depends on F3.1, F4.6. |
| F3.21  | Clear deck selection on withdrawal       | Medium   | When a participant **withdraws** from an event or **switches mode from playing to spectating** (F3.4), the system automatically clears their `EventDeckEntry` (played deck) for that event. This prevents ghost entries where a non-playing participant still appears as playing a deck. The cleared deck becomes available for others to select. A flash message confirms the action: "Your deck selection has been cleared." Depends on F3.4, F3.7. |

## F4 — Borrow Workflow

| ID     | Feature                                  | Priority | Description |
|--------|------------------------------------------|----------|-------------|
| F4.1   | Request to borrow a deck for an event    | High     | A participant requests a specific deck (or archetype preference) for a declared event. |
| F4.2   | Approve / deny borrow request            | High     | The deck owner reviews and approves or denies the request. |
| F4.3   | Confirm deck hand-off (lend)             | High     | At the event, the lend is confirmed — deck status changes to "lent". Can be performed by the owner or by event staff (see F4.8). Multiple identification paths: **(A) Player-first (QR)** — player shows QR → scan (F1.12) → app shows approved borrow → find deck → scan deck label (F5.3/F5.6) → confirmed. **(B) Deck-first (verbal)** — player says deck name or short tag → organizer finds & scans deck → app shows approved borrow with player info → verify identity → confirmed. **(C) Borrow list** — open borrow inbox (F4.10) or staff custody list (F4.9) → find borrow → scan deck → confirmed. **(D) Fully manual** — search borrow list → find borrow → manually confirm without scanning. |
| F4.4   | Confirm deck return                      | High     | After the event, the return is confirmed — deck status changes back to "available". Ideally done by scanning the deck label. Can be performed by the owner or by event staff (see F4.8). |

| F4.11  | Borrow conflict detection               | High     | Detect **temporal conflicts** when a deck is requested for overlapping events. **Hard block:** if the deck is `approved` or `lent` for an overlapping event, new requests and approvals are blocked. **Pending requests never block:** multiple players can request the same deck — pending borrows do not hide the deck from availability lists or prevent new requests. This lets the owner compare competing requests and choose the best attribution. **Auto-decline on approval:** when a borrow is approved or a walk-up lend is created, all other pending borrows for the same deck at the same event are automatically declined (via async Messenger handler), with the deck owner recorded as the actor. **Soft warning:** pending requests for overlapping events show a warning but the owner can still approve. Overlap rule: `event_A.date < event_B.endDate AND event_B.date < event_A.endDate`. See [Borrow model — Conflict Detection](models/borrow.md). |
| F4.12  | Walk-up lending (direct lend at event)  | High     | An owner, organizer, or event staff initiates a **direct lend** at the event without a prior borrow request. If the player has no account, the initiator asks them to self-register or performs a quick-create from QR (F1.12). The lending player (owner) must themselves be engaged in the event (spectating, interested, or participating). The borrow is created directly in `lent` state, skipping request/approval (`requestedAt`, `approvedAt`, and `handedOffAt` all set to now). Return flow identical to F4.4. See [Borrow model — Walk-up Path](models/borrow.md). Depends on F2.5, F3.5. |
| F4.13  | Event-scoped autocompletes              | Medium   | For **deck and player autocomplete** inputs in borrow-related flows (walk-up lending F4.12, staff delegation F4.8, borrow request F4.1), filter suggestions based on **event relationships**. This prevents selecting decks or players that have no relationship to the event, reducing errors at busy events. Depends on F4.1, F3.4. |
| F4.14  | Staff custody handover tracking         | Medium   | For **staff-delegated decks** (F4.8), register the actual **physical hand-over** from owner to staff and from staff back to owner as explicit tracked steps. Two new transitions: **(1) Owner → Staff**: owner confirms handing a delegated deck to a staff member (sets `staffReceivedAt` and identifies the receiving staff member). **(2) Staff → Owner**: staff confirms returning the deck to the owner after the event (sets `staffReturnedAt`). | F4.15  | Friend delegation for borrow completion | Low      | Allow a deck owner to **delegate borrow completion authority** to a trusted friend for a specific event. The delegated friend can approve/deny borrow requests and confirm hand-offs/returns on behalf of the owner, without requiring staff status. Useful when the owner cannot attend but wants a specific person (not necessarily event staff) to manage their decks. The delegation is per-owner, per-event (not per-deck). The friend must have an account and be a participant of the event. Depends on F4.2, F4.3, F3.4. |

## F12 — Quality & Security Consolidation

| ID      | Feature                                      | Priority | Description |
|---------|----------------------------------------------|----------|-------------|
| F12.1   | Controller role & context security audit     | High     | Audit **all controllers** to verify that every action is properly secured by: **(1) Role checks** — each endpoint requires the appropriate global role (`ROLE_ADMIN`, `ROLE_ORGANIZER`, `ROLE_PLAYER`) or per-event role (staff). **(2) Context checks** — actions on decks verify the current user is the deck owner (or has delegated authority). | F12.2   | Optimistic state conflict detection          | Medium   | For all **state-changing controller actions** (borrow approve/deny, hand-off, return, registration toggles, deck selection, event edits), detect when the entity's state has been modified by another user between the page load and the form submission. Implementation: include a **version token** (e.g. `updatedAt` timestamp or a dedicated version counter) as a hidden form field. On submission, compare the submitted token with the current entity state — if they differ, reject the action with a flash message ("This record was modified by another user. Please review and try again.") and redirect back to the current state. Prevents race conditions at busy events where multiple staff members may act on the same borrow simultaneously. |
